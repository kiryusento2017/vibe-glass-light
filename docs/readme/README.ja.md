## これは何

**Claude Code Light** は Windows デスクトップウィジェットです。**常に最前面に浮かぶ**液体ガラスのカプセルで、それ越しに実際のデスクトップがリアルタイムに屈折・歪んで見えます。カプセルの中央には 3 つの信号機が埋め込まれていて、**[Claude Code](https://claude.ai/code) の作業状態をリアルタイムに映し出します**——

> 赤の点滅 = ツール実行中；黄の点滅 = 思考中；緑の点灯 = アイドル（待機中）；全消灯 = 起動していない。

ターミナルをずっと見つめなくても、**視界の隅でウィジェットを一瞥すれば、AI が忙しいのか・考えているのか・止まっているのかが分かります**。単一のネイティブ `.exe` で、ランタイム依存なし、USB メモリに入れて差すだけで使えます。

## ✨ 設計思想

インスピレーションは **Apple の「Liquid Glass（液体ガラス）」**。背景にぼかしを一枚かぶせるだけの手法とは違い、これは**本物の屈折**です：

- **実際のデスクトップをリアルタイムに屈折** —— 画面のピクセルを自前で取得し、自作の HLSL 屈折シェーダーを通すことで、ガラスの後ろにあるウィンドウ・コード・壁紙が本物のガラス越しのように中心へ向かって歪みます（中心はクリア、縁ほど強く屈折）。静止画でも、ただのすりガラスのぼかしでもありません。
- **ドラッグ可能** —— カプセルを掴んで画面のどこへでも移動でき、離すと位置を記憶します。
- **押すとプルプル変形** —— ここが魂です 🍮。押し込むとガラスはゼリーのように**横に潰れ、縦に伸び**ます；速くドラッグするほど「振られて」細くなり；離すと**二次のバネ物理**でプルンと弾み、わずかにオーバーシュートしてから戻ります。変形パラメータ（剛性 / 減衰 / オーバーシュート）はすべて `glass-tuning.json` でホット調整でき、保存した瞬間に反映されます。
- **無段階スケーリング** —— 右クリック「サイズ変更」で 100%〜2000% まで自由に拡大縮小、サイズと位置は自動で記憶。

## 🚦 状態の対応表

| ランプ | 意味 | トリガーとなる Claude Code イベント |
|---|---|---|
| 🔴 **赤の点滅** | ツール実行中 | `PreToolUse` |
| 🟡 **黄の点滅** | 思考中 | `UserPromptSubmit` / `PostToolUse` |
| 🟢 **緑の点灯** | アイドル（待機中） | `Stop` |
| ⚫ **全消灯** | 未起動 | `claude.exe` プロセスが存在しない |

優先度：**赤 > 黄 > 緑 > グレー**。複数の Claude セッションが同時に走っている場合、どれか 1 つでも作業中なら「忙しい」と表示され、1 つが終わっただけで誤ってアイドル判定されることはありません。

## 💻 動作要件

| 要件 | 説明 |
|---|---|
| **OS** | **Windows 10 (2004+) / 11 のみ**。**macOS / Linux 非対応**——Windows 固有の DirectComposition + Desktop Duplication に依存しており、クロスプラットフォームの予定はありません。 |
| **Claude Code** | ローカルに [Claude Code](https://claude.ai/code) CLI がインストールされている必要があります。これは **Claude Code 専用**のステータスインジケーターで、Cursor / Copilot / その他の AI ツールには対応しません。未インストールでも起動はしますが、ランプは常にグレーのままです。 |
| **ランタイム** | **不要**。単一のネイティブ `.exe`。Node / .NET / Electron などのフレームワークに一切依存しません。 |

## 📦 インストール（3 ステップ）

**1️⃣ ダウンロード**

[Releases](https://github.com/kiryusento2017/vibe-glass-light/releases) から `claude-traffic-light.exe` をダウンロードし、任意のフォルダに置きます。

**2️⃣ 初回は管理者として実行**

> exe を右クリック →「管理者として実行」。**初回のみ必要**で、その後は通常どおりダブルクリックで OK。

初回起動時、exe と同じフォルダに 2 つの設定ファイルが作られます：`config.json`（位置 / スケール）と `glass-tuning.json`（外観 / 変形パラメータ）。

**3️⃣ 作られた 2 つのファイルに読み取り + 書き込み権限を付与**

ウィジェットが **C: ドライブ**（特に `C:\Program Files\` のような保護されたフォルダ）にある場合、設定ファイルの書き込みが OS に拒否されることがあります。その際はこの 2 ファイルに手動で**読み取り + 書き込み**権限を付けます：ファイルを右クリック →「プロパティ」→「セキュリティ」→「編集」→「書き込み」にチェック。

> 💡 **このステップが必要なのは基本的に C: ドライブの場合だけ**です。D: などの他ドライブに置けば、通常は管理者権限なしでそのまま設定ファイルを作れるので、スキップして構いません。

![config.json / glass-tuning.json に読み書き権限を付与](../../assets/admin-permission.png)

セットアップが終わったら、Claude Code を開いて vibe coding を始めれば、ランプが連動します。

## 🔒 あなたの PC のどのファイルに触れるのか？

見知らぬ `.exe` を動かすとき、一番気になるのは「自分の PC に何を書き込むのか」でしょう。ここでは、触れる**すべてのファイルと、何で動いているのか**を包み隠さず説明します——本ウィジェットは**システムファイルを一切変更せず、ネットワークにも一切接続しません**。

### 何で動くのか：Claude Code の「フック (Hooks)」

Claude Code には「フック」という仕組みが備わっています：**ツールの実行開始・思考・停止**などの重要なタイミングで、あらかじめ登録しておいたコマンドを自動実行するものです。本ウィジェットはこれを利用します——初回起動時に Claude Code 自身の設定ファイル `~/.claude/settings.json` に**4 つのフックを追加**し、状態が変わるたびに Claude Code がウィジェットへ「通知」してくれるようにします。その 4 つのフックが呼ぶコマンドは**ウィジェット自身**（`claude-traffic-light.exe hook <state>`）で、サードパーティ製プログラムには一切依存しません（Node のインストールが必要な手法とは違います）。

フックをマージする際は非常に控えめです：

- 変更前に `settings.json` 全体を `settings.json.bak` として**バックアップ**；
- **自分の 4 つだけを追加し、あなたの既存設定は一切削除・変更しません**；
- すでに追加済みなら重複して追加しません（冪等）；
- **Claude Code が未インストール（`~/.claude/` フォルダが存在しない）なら、何も書き込まず、フォルダも作りません。**

### 作成 / 書き込むファイル一覧

| ファイル | 場所 | 用途 | 書き込むタイミング |
|---|---|---|---|
| `config.json` | **exe と同じフォルダ** | 位置 / サイズ / 各種トグルを記憶 | ドラッグ・スケール変更・メニュー操作時 |
| `glass-tuning.json` | **exe と同じフォルダ** | ガラスの外観・変形パラメータ（手動調整用） | 初回起動時に一度生成 |
| `settings.json`（**4 つのフックを追加**） | `~/.claude/` | 状態変化時に Claude Code から通知させる | 初回起動時；変更前にバックアップ |
| `settings.json.bak` | `~/.claude/` | 上記の**変更前の完全バックアップ** | フックに変更がある時のみ |
| `agent-light-state-<セッションID>` | `~/.claude/agent-light/` | 各 Claude セッションの現在の状態（**1 単語だけ**：`running`/`thinking`/`idle`） | 状態が変わるたびに上書き |
| レジストリ `Run` 項目 | `HKCU\...\Run`（ユーザー領域、**UAC ポップアップなし**） | 「スタートアップ起動」の実現 | **チェックした場合のみ**；解除で削除 |

> 最初の 2 つ（`config.json` / `glass-tuning.json`）は exe のすぐ隣にあり、一目で見えていつでも削除できます。残りはすべてあなた自身の `~/.claude/` ユーザーフォルダ内です。**C: のシステムフォルダや、レジストリの重要領域には一切触れません。**

### 複数の Claude を同時に走らせても混乱しない

Claude セッションを 1 つ開くたびに、ウィジェットはそれ専用の**小さな状態ファイル**（中身は 1 単語）を作ります。複数同時に走るときは、ウィジェットが 0.1 秒ごとにこれらをまとめて集計します：**どれか 1 つでも作業中ならランプは「忙しい」、すべてアイドルになって初めて緑になります。** こうして、あるエージェントが先に終わって別のがまだ動いている場合でも、ランプが**誤って緑になることはありません**。これらの小さなファイルは 10 分以上更新されないと自動削除され、溜まり続けることはありません。

### 完全に消したい場合は？

exe と同じフォルダの 2 つの `.json` ＋ `~/.claude/agent-light/` フォルダを削除し、`~/.claude/settings.json` からあの 4 つのフックを削除（または `.bak` から復元）すればOK。「スタートアップ起動」をオフにすれば、レジストリ項目も自動で削除されます。**きれいさっぱり、痕跡を残しません。**

## 🎮 使い方

- **ドラッグ** —— 見えているカプセルを掴んで移動（カプセル外の透明な部分は反応しないので、誤ドラッグしません）。
- **右クリックメニュー / トレイメニュー**：
  | メニュー項目 | 機能 |
  |---|---|
  | 表示 / 非表示 | 表示を切り替え。非表示後はトレイアイコンから復帰 |
  | 位置を固定 | ロックするとドラッグ不可、誤操作による移動を防止 |
  | スタートアップ起動 | レジストリ Run 項目に書き込み（ユーザー領域、UAC なし）、解除で削除 |
  | サイズ変更… | スライダーウィンドウ、100%〜2000% の無段階スケーリング |
  | サイズと位置をリセット | 100%・画面上部中央へ戻す |
  | 再起動 | フリーズ / 異常時のワンクリック再起動（新インスタンスが引き継ぎ、「閉じたまま起動しない」ことは絶対にない） |
  | 終了 | — |
- **システムトレイ** —— トレイアイコンを右クリック = 同じメニュー；ダブルクリック = 表示 / 非表示。
- **パラメータ調整** —— exe と同じフォルダの `glass-tuning.json` を編集すれば、**保存した瞬間にホットリロード（〜0.5 秒で反映）**、再起動不要。

### レンダリングパイプライン（中核）

```
Desktop Duplication で画面全体のデスクトップテクスチャを取得（GPU 常駐）
  → HLSL のスーパー楕円 SDF + 屈折コア（中心はクリア、縁は強く屈折）
  → DXGI コンポジション swapchain → DirectComposition 透明最前面ウィンドウ
ウィンドウに WDA_EXCLUDEFROMCAPTURE を設定して自身をキャプチャから除外し、「自分が自分を屈折する」フィードバックを断つ
```

なぜ CSS `backdrop-filter` / WebView2 を使わないのか？それらは WebView ドキュメント内部しかサンプリングできず、**OS のデスクトップを取得できない**ため、旧版は黒い箱としてしか表示できませんでした。Windows にも「ウィンドウ背景を屈折変位させる」システム API は存在しません（DWM Acrylic はぼかしのみで屈折なし）。唯一の道は、デスクトップのピクセルを自前で取得し、自作の屈折シェーダーを書くことです。

### モジュール構成

```
main.go             エントリ：単一インスタンスのミューテックス → 設定読み込み → スタートアップ同期 → フック導入 → ウィンドウ + 監視
hookinstall.go      状態フックを ~/.claude/settings.json に冪等にマージ（先にバックアップ、追加のみ）
config/             config.json（位置/スケール/スタートアップ）+ glass-tuning.json（ビジュアルのホットリロード）
  autostart.go      レジストリ HKCU Run の読み書き削除 + パス自己補正
state/              4 状態列挙（グレー/緑/黄/赤）と優先度集約
watcher/            100ms ごとにセッション状態ファイルを集約（どれか忙=忙）+ 3s ごとのプロセス検出フォールバック
ui/
  window.go           DComp 透明最前面ウィンドウ、メッセージループ、独自ドラッグ、バネ変形ステートマシン
  render.go           D3D11 レンダリングパイプライン：device/swapchain/shader コンパイル + 毎フレーム描画
  glass.hlsl          ピクセルシェーダー：スーパー楕円 SDF で形状を制限 + 屈折コア + 3 ランプ
  capture.go          Desktop Duplication でデスクトップテクスチャ取得（スケールに応じ Resize + 無効化後にレート制限付き再構築）
  com.go              D3D11/DXGI/DComp の COM バインディング
  win32.go            Win32 API バインディングと定数
  physics.go          二次のバネ物理（毎フレーム Euler 積分で変形を駆動）
  sizedialog.go       サイズ変更スライダーウィンドウ（comctl32 trackbar）
```

### 状態検出：Claude Code Hooks（リアルタイム駆動）

transcript をポーリングするのではなく、Claude Code の 4 つのライフサイクルフックで状態を**リアルタイムにプッシュ**します。初回起動時、この 4 つを `~/.claude/settings.json` に冪等にマージします：

| Claude Code イベント | 書き込む状態語 | ランプ |
|---|---|---|
| `UserPromptSubmit` / `PostToolUse` | `thinking` | 🟡 思考中 |
| `PreToolUse` | `running` | 🔴 実行中 |
| `Stop` | `idle` | 🟢 アイドル |

各フックは exec 形式でウィジェット自身 `claude-traffic-light.exe hook <状態語>` を直接呼び出し（シェルを介さず直接 spawn、Windows のシェル不確定性を回避）、やることは 1 つだけ：状態語を**セッションごとに独立した状態ファイル**へ書き込むこと。

#### セッションごとに 1 つの状態ファイル `agent-light-state-<session_id>`

Claude Code はフックを発火する際、**stdin で JSON を渡し**、その中に現在のセッションの `session_id` が含まれます。ウィジェットはそれを読み取り、次へ書き込みます：

```
~/.claude/agent-light/agent-light-state-<session_id>
```

ファイルの中身は状態語 1 つ（`running` / `thinking` / `idle`）。ポイント：

- **セッションごとに 1 ファイル、互いに上書きしない**——これがマルチエージェント対応の土台です（後述）。
- session_id は安全なファイル名にフィルタされます（英数字 / ハイフンのみ）；取得できない場合は `default` にフォールバック。
- stdin の読み取りには **500ms のハードタイムアウト**があり、フックは常にミリ秒で返り、**Claude Code を遅くすることは決してありません**。
- すべて `agent-light/` サブフォルダにまとめ、`~/.claude/` 直下にファイルを散らかしません。

#### マルチエージェント並行：どれか 1 つでも忙=全体が忙

**複数の Claude を同時に走らせる**こと（複数ターミナル、またはメインセッションがサブエージェントを派生）は十分にあり得て、それぞれが自分の状態ファイルを書きます。ウィジェットの watcher は 100ms ごとに `agent-light/` 内のすべての `agent-light-state-*` ファイルを読み、優先度 **赤 > 黄 > 緑 > グレー** で 1 つの全体状態に集約します：

```
セッション A: running  🔴 ┐
セッション B: idle     🟢 ├─ 最高優先度を採用 ─→ 🔴 赤（A がまだ作業中なので忙しいと表示）
セッション C: thinking 🟡 ┘
```

**どれか 1 つでも作業中なら、ウィジェットは忙しいと表示します。** こうして、あるエージェントが先に終わって（`idle` を書き）別のがまだ動いているとき、ランプが**誤って緑になることはありません**；全セッションが `idle` で初めて緑、ファイルが 1 つも無いときだけグレーになります。

#### その他のフォールバック

- **忙しい状態はフックの内容だけを信頼**し、時間窓によるタイムアウト降格はしません（長考はツール呼び出しがなく → フックもないため、タイムアウトだと誤ってアイドル判定される）。
- **プロセス検出フォールバック**：3s ごとに `CreateToolhelp32Snapshot` で `claude.exe` を確認し、無ければ強制グレー——クラッシュ / 強制終了 / 起動時の残留はすべてこれでグレーに戻ります。
- **定期クリーンアップ**：30s ごとに 10 分以上更新されていない残留セッションファイルを削除（純粋なディスク回収で、状態判定とは独立）。
- **フックハンドラはウィジェット自身**で、外部依存ゼロ（node が必要な手法とは違います）。

Go ツールチェーン + Windows が必要：

```powershell
# デバッグ（コンソール出力あり、exe は生成しない）

# テスト
go test ./...
```

> **ビルドの鉄則**：デバッグは `go run .`；exe を作るときは上記の 1 行のみ——`-trimpath`（ローカルパス / ユーザー名を除去）、`-buildvcs=false`（git 情報を除去）、`-ldflags="-H windowsgui"`（黒いコンソールなし）、`-o dist/`（隔離）、そして `rsrc_windows_amd64.syso`（アイコン + バージョン情報）を自動埋め込み。**`-s -w` は厳禁**（アンチウイルスの誤検知を誘発）、**パッキングも厳禁**。

**録画デモ**：通常の exe は OBS などの録画ソフトから不可視です（`WDA_EXCLUDEFROMCAPTURE` による設計）。`--demo` 付きで起動すれば除外を解除して OBS で録れますが、その場合ガラスが自分自身を屈折します。最もきれいなデモ映像が欲しいなら、スマホ / カメラで画面を撮るのが手っ取り早いです。
