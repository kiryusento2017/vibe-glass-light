## 이게 뭔가요

**Glight**(별칭 Claude Code Light / Claude Code 신호등 상태 표시기)는 Windows 바탕화면 위젯입니다. **항상 최상단에 떠 있는** 리퀴드 글래스 캡슐로, 그 너머로 실제 바탕화면이 실시간으로 굴절되어 일그러져 보입니다. 캡슐 가운데에는 세 개의 신호등이 박혀 있어 **[Claude Code](https://claude.ai/code)의 작업 상태를 실시간으로 보여줍니다**——

> 빨강 점멸 = 도구 실행 중; 노랑 점멸 = 생각 중; 초록 켜짐 = 대기(유휴); 모두 꺼짐 = 실행 중 아님.

터미널을 계속 쳐다볼 필요 없이, **곁눈으로 위젯을 한 번 흘끗 보면 AI가 바쁜지, 생각 중인지, 멈췄는지 알 수 있습니다**. 단일 네이티브 `.exe`이며, 런타임 의존성이 없어 USB에 넣어 꽂기만 하면 바로 동작합니다.

## ✨ 설계 철학

영감은 **Apple의 "Liquid Glass(리퀴드 글래스)"**에서 왔습니다. 배경에 블러 한 겹을 덧씌우는 방식과 달리, 이것은 **진짜 굴절**입니다:

- **실제 바탕화면을 실시간 굴절** —— 화면 픽셀을 직접 가져와 자체 제작 HLSL 굴절 셰이더를 통과시켜, 유리 뒤의 창·코드·배경화면이 진짜 유리 너머처럼 중심을 향해 휘어집니다(중심은 선명, 가장자리일수록 강하게 굴절). 정지 이미지도, 단순한 불투명 블러도 아닙니다.
- **드래그 가능** —— 캡슐을 잡고 화면 어디로든 옮길 수 있고, 놓으면 위치를 기억합니다.
- **누르면 말랑말랑 변형** —— 여기가 핵심입니다 🍮. 누르면 유리가 젤리처럼 **가로로 눌리고 세로로 늘어납니다**; 빠르게 드래그할수록 "휘둘려" 가늘어지고; 놓으면 **2차 스프링 물리**로 탱글탱글 튀어 오르며 살짝 오버슈트한 뒤 제자리로 돌아옵니다. 변형 파라미터(강성 / 감쇠 / 오버슈트)는 모두 `glass-tuning.json`에서 핫 조정할 수 있으며 저장 즉시 반영됩니다.
- **무단계 스케일링** —— 우클릭 "크기 조정"으로 100%~2000% 자유 확대·축소, 크기와 위치는 자동 기억.

## 🚦 상태 대조표

| 램프 | 의미 | 트리거되는 Claude Code 이벤트 |
|---|---|---|
| 🔴 **빨강 점멸** | 도구 실행 중 | `PreToolUse` |
| 🟡 **노랑 점멸** | 생각 중 | `UserPromptSubmit` / `PostToolUse` |
| 🟢 **초록 켜짐** | 대기(유휴) | `Stop` |
| ⚫ **모두 꺼짐** | 실행 중 아님 | `claude.exe` 프로세스 없음 |

우선순위: **빨강 > 노랑 > 초록 > 회색**. 여러 Claude 세션이 동시에 돌아갈 때, 하나라도 작업 중이면 "바쁨"으로 표시되며, 하나가 끝났다고 해서 잘못 유휴로 판정하지 않습니다.

## 💻 시스템 요구 사항

| 요구 사항 | 설명 |
|---|---|
| **OS** | **Windows 10 (2004+) / 11 전용**. **macOS / Linux 미지원**——Windows 고유의 DirectComposition + Desktop Duplication에 의존하며, 크로스 플랫폼 계획은 없습니다. |
| **Claude Code** | 로컬에 [Claude Code](https://claude.ai/code) CLI가 설치되어 있어야 합니다. 이것은 **Claude Code 전용** 상태 표시기로, Cursor / Copilot / 기타 AI 도구는 지원하지 않습니다. 미설치 시에도 실행은 되지만 램프는 계속 회색입니다. |
| **런타임** | **불필요**. 단일 네이티브 `.exe`. Node / .NET / Electron 등 어떤 프레임워크에도 의존하지 않습니다. |

## 📦 설치 (3단계)

**1️⃣ 다운로드**

[Releases](https://github.com/kiryusento2017/Glight/releases)에서 `claude-traffic-light.exe`를 받아 아무 폴더에나 둡니다.

**2️⃣ 최초 실행은 관리자 권한으로**

> exe 우클릭 → "관리자 권한으로 실행". **최초 한 번만 필요**하며, 이후에는 평소처럼 더블클릭하면 됩니다.

최초 실행 시 exe와 같은 폴더에 설정 파일 두 개가 생성됩니다: `config.json`(위치 / 배율)과 `glass-tuning.json`(외관 / 변형 파라미터).

**3️⃣ 생성된 두 파일에 읽기 + 쓰기 권한 부여**

위젯이 **C: 드라이브**(특히 `C:\Program Files\` 같은 보호된 폴더)에 있으면, OS가 설정 파일 쓰기를 막을 수 있습니다. 이때 두 파일에 수동으로 **읽기 + 쓰기** 권한을 줍니다: 파일 우클릭 → "속성" → "보안" → "편집" → "쓰기" 체크.

> 💡 **이 단계는 보통 C: 드라이브일 때만 필요**합니다. D: 등 다른 드라이브에 두면 대개 관리자 권한 없이도 설정 파일을 바로 생성할 수 있으니 건너뛰어도 됩니다.

![config.json / glass-tuning.json 에 읽기·쓰기 권한 부여](../../assets/admin-permission.png)

설정이 끝나면 Claude Code를 열고 vibe coding을 시작하세요. 램프가 따라 움직입니다.

## 🔒 내 컴퓨터의 어떤 파일을 건드리나요?

낯선 `.exe`를 실행할 때 가장 신경 쓰이는 건 "내 컴퓨터에 뭘 쓰는가"일 겁니다. 여기서는 건드리는 **모든 파일과 무엇으로 동작하는지**를 전부 투명하게 설명합니다——이 위젯은 **어떤 시스템 파일도 변경하지 않으며, 네트워크에도 전혀 연결하지 않습니다**.

### 무엇으로 동작하나: Claude Code의 "훅(Hooks)"

Claude Code에는 "훅"이라는 메커니즘이 내장되어 있습니다: **도구 실행 시작·생각·정지** 같은 핵심 순간에, 미리 등록해 둔 명령을 자동 실행합니다. 이 위젯은 그 메커니즘을 이용합니다——최초 실행 시 Claude Code 자체 설정 파일 `~/.claude/settings.json`에 **훅 4개를 추가**하여, 상태가 바뀔 때마다 Claude Code가 위젯에게 "알려" 주도록 합니다. 이 4개의 훅이 호출하는 명령은 **위젯 자신**(`claude-traffic-light.exe hook <state>`)이며, 어떤 서드파티 프로그램에도 의존하지 않습니다(Node 설치가 필요한 방식과는 다릅니다).

훅을 병합할 때는 매우 신중합니다:

- 변경 전에 `settings.json` 전체를 `settings.json.bak`으로 **백업**;
- **자신의 4개만 추가하고, 기존 설정은 절대 삭제·수정하지 않습니다**;
- 이미 추가됐으면 중복 추가하지 않습니다(멱등);
- **Claude Code가 미설치(`~/.claude/` 폴더 없음)라면 아무것도 쓰지 않고, 폴더도 만들지 않습니다.**

### 생성 / 기록하는 전체 파일

| 파일 | 위치 | 용도 | 기록 시점 |
|---|---|---|---|
| `config.json` | **exe와 같은 폴더** | 위젯의 위치 / 크기 / 각종 토글 기억 | 드래그·크기 변경·메뉴 조작 시 |
| `glass-tuning.json` | **exe와 같은 폴더** | 유리 외관·변형 파라미터(수동 조정용) | 최초 실행 시 한 번 생성 |
| `settings.json`(**훅 4개 추가**) | `~/.claude/` | 상태 변화 시 Claude Code가 위젯에 통지하도록 | 최초 실행 시; 변경 전 백업 |
| `settings.json.bak` | `~/.claude/` | 위 파일의 **변경 전 전체 백업** | 훅에 변동이 있을 때만 |
| `agent-light-state-<세션ID>` | `~/.claude/agent-light/` | 각 Claude 세션의 현재 상태(**단어 하나**: `running`/`thinking`/`idle`) | 상태가 바뀔 때마다 덮어쓰기 |
| 레지스트리 `Run` 항목 | `HKCU\...\Run`(사용자 영역, **UAC 팝업 없음**) | "부팅 시 자동 시작" 구현 | **체크한 경우에만**; 해제하면 삭제 |

> 앞의 두 개(`config.json` / `glass-tuning.json`)는 exe 바로 옆에 있어 한눈에 보이고 언제든 삭제할 수 있습니다. 나머지는 모두 본인의 `~/.claude/` 사용자 폴더 안에 있습니다. **C: 시스템 폴더나 레지스트리 민감 영역은 전혀 건드리지 않습니다.**

### 여러 Claude를 동시에 돌려도 꼬이지 않습니다

Claude 세션을 하나 열 때마다 위젯은 그 세션 전용 **작은 상태 파일**(내용은 단어 하나)을 만듭니다. 여러 개가 동시에 돌아갈 때 위젯은 0.1초마다 이 파일들을 한 번씩 모아 집계합니다: **하나라도 작업 중이면 램프는 바쁨, 모두 유휴가 되어야 비로소 초록이 됩니다.** 이렇게 하나의 에이전트가 먼저 끝나고 다른 게 아직 돌아갈 때도 램프가 **잘못 초록으로 바뀌지 않습니다**. 이 작은 파일들은 10분 넘게 갱신되지 않으면 자동 정리되어 쌓이지 않습니다.

### 완전히 지우고 싶다면?

exe와 같은 폴더의 두 `.json` + `~/.claude/agent-light/` 폴더를 삭제하고, `~/.claude/settings.json`에서 그 훅 4개를 지우면(또는 `.bak`으로 복원하면) 됩니다. "부팅 시 자동 시작"을 끄면 레지스트리 항목도 자동 삭제됩니다. **깔끔하게 제거되어 흔적을 남기지 않습니다.**

## 🎮 사용법

- **드래그** —— 보이는 캡슐을 잡고 옮깁니다(캡슐 밖 투명 영역은 반응하지 않아 잘못 드래그되지 않습니다).
- **우클릭 메뉴 / 트레이 메뉴**:
  | 메뉴 항목 | 기능 |
  |---|---|
  | 표시 / 숨김 | 표시 전환, 숨긴 후 트레이 아이콘에서 복원 |
  | 위치 고정 | 잠그면 드래그 불가, 실수로 이동 방지 |
  | 부팅 시 자동 시작 | 레지스트리 Run 항목에 기록(사용자 영역, UAC 없음), 해제하면 삭제 |
  | 크기 조정… | 슬라이더 창, 100%~2000% 무단계 스케일링 |
  | 크기·위치 초기화 | 100%·화면 상단 중앙으로 복귀 |
  | 재시작 | 멈춤 / 이상 시 원클릭 재시작(새 인스턴스가 인계, "끈 채로 안 켜지는" 일은 절대 없음) |
  | 종료 | — |
- **시스템 트레이** —— 트레이 아이콘 우클릭 = 동일 메뉴; 더블클릭 = 표시 / 숨김.
- **파라미터 조정** —— exe와 같은 폴더의 `glass-tuning.json`을 편집하면 **저장 즉시 핫 리로드(~0.5초 내 반영)**, 재시작 불필요.

| 파라미터 | 효과 | 기본 | 권장 범위 |
|---|---|---|---|

### 렌더링 파이프라인(핵심)

```
Desktop Duplication으로 전체 화면 바탕화면 텍스처 획득(GPU 상주)
  → HLSL 슈퍼타원 SDF + 굴절 코어(중심 선명, 가장자리 강한 굴절)
  → DXGI 컴포지션 swapchain → DirectComposition 투명 최상단 창
창에 WDA_EXCLUDEFROMCAPTURE를 설정해 자신을 캡처에서 제외, "자기 자신을 굴절하는" 피드백 차단
```

왜 CSS `backdrop-filter` / WebView2를 안 쓰나? 그것들은 WebView 문서 내부만 샘플링할 수 있어 **OS 바탕화면을 가져올 수 없어서** 구버전은 검은 상자로만 표시됐습니다. Windows에도 "창 배경을 굴절 변위시키는" 시스템 API는 없습니다(DWM Acrylic은 블러만 있고 굴절 없음). 유일한 길은 바탕화면 픽셀을 직접 가져와 굴절 셰이더를 직접 작성하는 것입니다.

### 상태 검출: Claude Code Hooks(실시간 구동)

transcript를 폴링하지 않고 Claude Code의 4개 라이프사이클 훅으로 상태를 **실시간 푸시**합니다. 최초 실행 시 이 4개를 `~/.claude/settings.json`에 멱등 병합합니다:

| Claude Code 이벤트 | 기록 상태어 | 램프 |
|---|---|---|
| `UserPromptSubmit` / `PostToolUse` | `thinking` | 🟡 생각 중 |
| `PreToolUse` | `running` | 🔴 실행 중 |
| `Stop` | `idle` | 🟢 유휴 |

각 훅은 exec 형식으로 위젯 자신 `claude-traffic-light.exe hook <상태어>`를 직접 호출하고(셸을 거치지 않고 직접 spawn, Windows 셸 불확정성 회피), 하는 일은 단 하나: 상태어를 **세션별 독립 상태 파일**에 기록하는 것입니다.

#### 세션마다 하나의 상태 파일 `agent-light-state-<session_id>`

Claude Code는 훅을 발동할 때 **stdin으로 JSON을 전달**하며, 그 안에 현재 세션의 `session_id`가 들어 있습니다. 위젯은 이를 읽어 다음에 기록합니다:

```
~/.claude/agent-light/agent-light-state-<session_id>
```

파일 내용은 상태어 하나(`running` / `thinking` / `idle`). 핵심:

- **세션마다 한 파일, 서로 덮어쓰지 않음**——이것이 멀티 에이전트 지원의 토대입니다(아래 참조).
- session_id는 안전한 파일명으로 필터링됩니다(영숫자 / 하이픈만); 가져오지 못하면 `default`로 폴백.
- stdin 읽기에는 **500ms 하드 타임아웃**이 있어 훅은 항상 밀리초 안에 반환하며 **Claude Code를 느리게 하지 않습니다**.
- 모두 `agent-light/` 하위 폴더에 모아 `~/.claude/` 최상위에 파일을 흩뿌리지 않습니다.

#### 멀티 에이전트 동시성: 하나라도 바쁨 → 전체 바쁨

**여러 Claude를 동시에 돌리는 것**(여러 터미널, 또는 메인 세션이 서브 에이전트를 파생)은 충분히 있을 수 있고, 각자 자기 상태 파일을 씁니다. 위젯의 watcher는 100ms마다 `agent-light/` 내 모든 `agent-light-state-*` 파일을 읽어 우선순위 **빨강 > 노랑 > 초록 > 회색**으로 하나의 전역 상태로 집계합니다:

```
세션 A: running  🔴 ┐
세션 B: idle     🟢 ├─ 최고 우선순위 채택 ─→ 🔴 빨강(A가 아직 작업 중이라 바쁨 표시)
세션 C: thinking 🟡 ┘
```

**하나라도 작업 중이면 위젯은 바쁨을 표시합니다.** 그래서 한 에이전트가 먼저 끝나(`idle` 기록) 다른 게 아직 돌아갈 때, 램프가 **잘못 초록으로 바뀌지 않습니다**; 모든 세션이 `idle`이어야 초록, 파일이 하나도 없을 때만 회색이 됩니다.

#### 그 밖의 폴백

- **바쁨 상태는 훅 내용만 신뢰**하고 시간 창 타임아웃 강등은 하지 않습니다(긴 사고는 도구 호출이 없어 → 훅도 없으므로, 타임아웃이면 잘못 유휴로 판정됨).
- **프로세스 검출 폴백**: 3s마다 `CreateToolhelp32Snapshot`로 `claude.exe`를 확인하고, 없으면 강제 회색——크래시 / 강제 종료 / 부팅 잔류는 모두 이걸로 회색 복귀.
- **정기 정리**: 30s마다 10분 넘게 갱신되지 않은 잔류 세션 파일을 삭제(순수 디스크 회수, 상태 판정과 분리).
- **훅 핸들러가 곧 위젯 자신**이라 외부 의존성 제로(node가 필요한 방식과 다름).

Go 툴체인 + Windows 필요:

```powershell
# 디버그(콘솔 출력 있음, exe 생성 안 함)
go run .

# exe 빌드(유일한 명령, 4가지 보호를 한 번에 적용)
go build -trimpath -buildvcs=false -ldflags="-H windowsgui" -o dist/claude-traffic-light.exe .

# 테스트
go test ./...
```

> **빌드 철칙**: 디버그는 `go run .`; exe를 만들 때는 위 한 줄만——`-trimpath`(로컬 경로 / 사용자명 제거), `-buildvcs=false`(git 정보 제거), `-ldflags="-H windowsgui"`(검은 콘솔 없음), `-o dist/`(격리), 그리고 `rsrc_windows_amd64.syso`(아이콘 + 버전 정보) 자동 임베드. **`-s -w` 금지**(백신 오탐 유발), **패킹 금지**.

**녹화 데모**: 일반 exe는 OBS 등 녹화 소프트웨어에 보이지 않습니다(`WDA_EXCLUDEFROMCAPTURE` 설계 때문). `--demo`로 실행하면 제외를 풀어 OBS로 녹화할 수 있지만, 그러면 유리가 자기 자신을 굴절합니다. 가장 깨끗한 데모 영상을 원하면 휴대폰 / 카메라로 화면을 찍는 게 가장 간단합니다.
